# Post-Mortem: Pre-rendered Pages Returning 404 on SWA

**Date:** 2026-06-16
**Scope:** Hybrid rendering support in `@opsydyn/astro-azure-swa`. Pre-rendered pages (pages with `export const prerender = true`) returned HTTP 404 on Azure Static Web Apps while rendering correctly locally.

---

## Summary

The `staticwebapp.config.json` generated by the adapter contained a `/*` catch-all rewrite rule that forwarded every request to the Azure Function. Azure SWA applies route rules before checking for static files, so even when a pre-rendered page's `index.html` existed in the deployment, SWA sent the request to the function instead of serving the file. The function rendered the page server-side and returned the correct HTML, but for pre-rendered content that meant unnecessary cold-start latency and â€” when the function hadn't yet warmed up â€” a 404.

The fix replaces the `/*` rewrite with SWA's `navigationFallback`, which is the platform-native mechanism for SSR fallback: SWA serves a static file if one exists, and only calls the fallback (the function) when no file is found.

---

## Issues

### 1. `/*` Route Rewrite Shadows All Static Files

**What happened:** The generated `staticwebapp.config.json` contained:

```json
{
  "routes": [
    { "route": "/_astro/*", "headers": { "cache-control": "..." } },
    { "route": "/*",        "rewrite": "/api/server" }
  ]
}
```

In SWA's routing pipeline, route rules are evaluated before static file lookup. A rule with a `rewrite` is not a conditional â€” it applies unconditionally to every matching path. For a pre-rendered page like `/hybrid`, the sequence was:

1. Request `GET /hybrid` arrives
2. SWA evaluates route rules: `/*` matches
3. SWA rewrites to `/api/server` and calls the function
4. `dist/client/hybrid/index.html` is **never consulted**

The page exists on the CDN; SWA simply never gets to check for it.

**Why it wasn't caught sooner:** Locally, `astro preview` uses Astro's own preview server which handles routing differently â€” it checks for static files first, then falls back to SSR. The routing discrepancy between local preview and SWA's edge is invisible during development.

**Fix:** Replaced the `/*` route rule with SWA's `navigationFallback`:

```json
{
  "routes": [
    { "route": "/_astro/*", "headers": { "cache-control": "..." } },
    { "route": "/",         "rewrite": "/api/server" }
  ],
  "navigationFallback": {
    "rewrite": "/api/server",
    "exclude": ["/_astro/*"]
  }
}
```

`navigationFallback` is a last-resort: SWA only invokes it when no static file is found at the requested path. Pre-rendered pages are found and served from the CDN; dynamic SSR routes that have no corresponding file go to the function.

**Lesson:** `routes` rewrites in SWA are unconditional. For an SSR adapter that also supports pre-rendered pages, `navigationFallback` is the correct mechanism. The two are not interchangeable: `routes` rewrites override static files, `navigationFallback` does not.

---

### 2. The Placeholder `index.html` Creates a Static Root

**What happened:** After switching to `navigationFallback`, a new subtlety appeared: the adapter writes an empty placeholder `index.html` to `dist/client/` so the SWA deploy action passes its "default file" validation check. With `navigationFallback`, SWA would find this file for `GET /` and serve an empty page instead of calling the function to server-render the homepage.

**Fix:** An explicit route rule for `/` is added alongside `navigationFallback`:

```json
{ "route": "/", "rewrite": "/api/server" }
```

Route rules take precedence over static file lookup, so the placeholder is never served. All other pre-rendered paths (e.g. `/hybrid`) fall through to static file serving via `navigationFallback`.

**Lesson:** The placeholder `index.html` exists to satisfy a deploy-time tooling check, not to be served to users. Any routing scheme that can accidentally serve it to users requires an explicit override.

---

### 3. User-Provided `/*` Override Must Suppress Generated Fallback

**What happened:** A user who wants a `/*` route (e.g. to serve a maintenance page for all traffic) should take full ownership of routing. If the adapter still emitted `navigationFallback` and a generated `/` route alongside the user's `/*`, both would be unreachable and the config would be misleading.

**Fix:** When the user provides a `/*` entry in `staticWebAppConfig.routes`, the adapter suppresses both the generated `/` route and `navigationFallback`. The user's `/*` rule already handles all paths â€” adding a fallback that can never trigger is noise.

**Lesson:** Generated routing config must be aware of what the user has already specified. A `/*` override is a signal of intent to own all routing; anything the adapter would add behind it is dead config.

## Mitigations for Next Time

| Risk | Mitigation |
|---|---|
| Route rule shadows static files | Use `navigationFallback` for SSR adapters; reserve `routes` rewrites for explicit path overrides |
| Routing differs between local preview and SWA | Add a deploy-and-verify step for hybrid routes in CI smoke test |
| Placeholder file accidentally served | Always pair a placeholder with an explicit route rule that overrides it |
| User `/*` override broken by generated config | Check for user-provided `/*` and suppress unreachable generated config |
